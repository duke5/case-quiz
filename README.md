# 病例答题比赛系统

大屏幕展示 + 主持人控制台(密码保护) + 手机答题端,题库单独存放在 `questions.md`,可随时编辑维护。大屏幕本身不设密码,只能从主持人登录后打开。

## 一、本地运行(先在自己电脑测试)

需要先安装 [Node.js](https://nodejs.org/)(选 LTS 版本,免费,安装后自带 npm)。

```bash
# 1. 进入项目目录
cd quiz-app

# 2. 安装依赖(只需第一次运行)
npm install

# 3. 启动服务
npm start
```

看到类似下面的提示就说明启动成功:

```
病例答题比赛服务已启动
本机访问: http://localhost:3000
局域网内其他设备访问: http://<你的电脑局域网IP>:3000
主持人密码: host2024
```

在浏览器打开 `http://localhost:3000` 即可看到**主持人登录页**(根路径现在直接就是主持人入口,不再有角色选择页)。登录页右上角有一个"打开大屏幕"按钮,点击会在新标签页打开大屏幕——大屏幕本身不再需要单独密码,访问权限靠主持人密码把关。

参赛者不通过这个页面进入——他们是扫大屏幕上自动显示的二维码,直接跳转到 `/player.html` 答题页,不会看到主持人登录界面。

## 二、活动当天怎么用(局域网内,不需要联网/不需要花钱)

1. 主持人电脑运行 `npm start`,保持这台电脑和参赛者手机在**同一个 WiFi**下
2. 查看这台电脑的局域网 IP(Windows 用 `ipconfig`,Mac 用 `ifconfig` 或系统设置里查看,类似 `192.168.1.100`),打开 `http://localhost:3000` 用主持人密码登录,在"参赛入口网址"里填 `http://192.168.1.100:3000` 并点更新(参见下面第四点五节,不用改文件)
3. 大屏幕电脑浏览器打开 `http://192.168.1.100:3000/screen.html`,不需要密码,直接就能看到——比赛开始前会**自动显示二维码**,参赛者直接扫码即可进入答题页,不需要再用第三方工具单独生成二维码(大屏幕的URL不建议随意公开,避免无关人员打开;正常情况下也是通过主持人登录后点按钮打开)
4. 主持人自己电脑/平板打开 `http://192.168.1.100:3000`(根路径就是主持人入口),输入主持人密码;也可以直接在这个页面点右上角"打开大屏幕"按钮,新标签页打开大屏幕
5. 参赛者手机扫大屏幕上的二维码 → 进入 `player.html` → 输入姓名进入

> 200 人在同一个路由器/WiFi 下没问题,如果现场 WiFi 承载能力存疑,建议提前用20-30台手机做一次压力测试。

## 三、如果想在互联网上用(比如参赛者不在同一WiFi,或想提前测试异地协作)

可以免费部署到 [Render](https://render.com)、[Railway](https://railway.app) 等平台(有免费额度,一次性活动完全够用):

1. 把整个 `quiz-app` 文件夹上传到 GitHub 仓库
2. 在 Render 新建一个 Web Service,连接这个仓库,Build 命令填 `npm install`,启动命令填 `npm start`
3. 部署成功后,**Render 会在服务详情页顶部直接显示分配给你的网址**,格式类似 `https://你的服务名-随机后缀.onrender.com`,创建服务的时候就能看到,不用等部署完成
4. 拿到这个网址后,**不需要改代码、不需要重新部署**——直接打开网站根路径(主持人登录页),登录后在"参赛入口网址"那一栏,把 Render 给你的网址粘贴进去,点"更新并重新生成二维码",大屏幕的二维码会立刻自动刷新成新地址
5. 密码记得通过环境变量 `HOST_PASSWORD` 覆盖成你自己的(不要用默认值),在 Render 的 Environment 设置里添加
6. 参赛者扫大屏幕上的二维码即可,不需要在同一WiFi下

> 关于"网址会不会自动同步"的说明:主持人控制台里更新的网址会保存在服务器上的 `config.local.json` 文件里(不是改 `config.js` 本身),重启服务/重新部署后这个文件**在 Render 免费版上不保证一定保留**(免费版偶尔会重建容器)。如果重启后发现二维码变回了旧地址,回到主持人控制台重新粘贴一次网址、点更新即可,不影响正在进行的比赛数据(成绩不受影响,只是二维码指向的地址需要重新填一下)。如果想要一个"重启也不会丢"的默认值,把最终确定的网址也同步写进 `config.js` 里的 `SITE_URL` 字段,提交到 GitHub 仓库、让 Render 用这个新代码重新部署一次即可长期生效。

## 四、怎么修改题目

直接编辑 `questions.md`,格式为:

```
## 题号(仅供人工阅读,不影响顺序)
题干文字
A. 选项A
B. 选项B
C. 选项C
D. 选项D
答案: 对应字母(A/B/C/D)
```

改完保存后:
- 如果服务还没启动:直接 `npm start` 即可生效
- 如果服务已经在运行:在主持人控制台点"重新加载题库"按钮,不用重启服务

## 四点五、怎么修改机构名称和参赛二维码指向的网址

**改参赛入口网址(SITE_URL),现在有两种方式:**

**方式一(推荐,活动当天用这个):** 主持人登录后,控制台顶部有"参赛入口网址"输入框,直接粘贴新地址、点"更新并重新生成二维码"——**立即生效,不用重启服务,不用碰任何文件**。这是专门为了应对"局域网IP每次都不一样"、"Render分配的域名要等部署完才知道"这些现场情况设计的。

**方式二(改机构名称/活动名称,或者想设置一个重启后也不会丢的默认网址):** 直接编辑根目录下的 `config.js`:

```js
module.exports = {
  ORG_NAME: '河北医科大学第一医院',   // 显示在首页/大屏幕/浏览器标签页标题
  EVENT_NAME: '病例答题比赛',
  SITE_URL: 'http://192.168.1.100:3000', // 参赛者手机扫码后会跳转到这个网址(默认值,方式一改过之后会被覆盖)
  SCORE_BASE: 50,
  SCORE_SPEED_BONUS_PER_SEC: 5,
};
```

改完保存后,在主持人控制台点"重新加载配置"按钮即可生效。

机构名称、活动名称只能通过方式二(改 `config.js`)修改,暂时没有做进管理页面——如果这两项也需要活动当天临时改,告诉我可以继续加。

大屏幕在"未开始"状态时会自动显示二维码,方便参赛者直接扫码进入,不需要额外用第三方工具生成二维码图片。

## 五、修改密码

打开 `server.js`,找到这一行改成你自己的密码:

```js
const HOST_PASSWORD = process.env.HOST_PASSWORD || 'host2024';
```

大屏幕不再需要单独设密码,只能通过主持人登录后点"打开大屏幕"按钮访问,或者知道 `/screen.html` 这个具体地址的人才能直接打开——正式活动建议只把这个地址告诉负责操作大屏的人。

也可以不改代码,启动时用环境变量覆盖(推荐,尤其是部署到公网时):

```bash
HOST_PASSWORD=你的主持人密码 npm start
```

## 六、计分规则(可自行调整)

答对得分 = `SCORE_BASE`(基础分) + 剩余时间(秒) × `SCORE_SPEED_BONUS_PER_SEC`(速度加成),答错得 0 分。默认是 50 分基础分 + 每秒5分速度加成。

这两个数字在 `config.js` 里,改完保存、主持人控制台点"重新加载配置"就生效,大屏幕底部的说明文字会自动跟着变(详见第九点七节)。如果想改的是整个计分方式本身(不只是这两个数字),需要改 `server.js` 里 `player:answer` 那部分的代码。

## 七、成绩持久化

比赛过程中成绩会实时写入 `scores.json`,即使服务意外重启,重新启动后也能恢复之前的成绩。如果想清空重新开始一场新比赛,主持人控制台有"重置比赛"按钮,或者直接删除 `scores.json` 文件。

## 八、关于身份识别(头像/姓名)

当前版本参赛者进入时手动填写姓名。如果后续想接入微信小程序自动获取头像昵称(2次点击确认的"头像昵称填写能力"),这部分需要额外开发微信小程序端并申请小程序账号(免费),可以在这个后端基础上扩展,把 `player:join` 换成小程序传来的真实昵称即可,后端逻辑不需要大改。

## 九点五、这次修复的问题(供参考)

- **断线重连会丢分的bug(重要)**:参赛者身份原来用 `socket.id`,微信内置浏览器切网络/切后台很容易断线重连,一重连 `socket.id` 就变了,等于变成"新玩家"、之前分数全丢。现在改成客户端生成一个持久化ID存在手机浏览器本地,断线重连、切后台、刷新页面都不会丢分,断线重连后会自动重新加入,不需要用户手动操作
- **重新加载配置后二维码/机构名可能显示旧内容**:`config.js` 和二维码接口现在都加了不缓存的响应头
- **重新加载题库把题目改少后可能导致后台越界访问**:如果当前题号超出新题库范围,自动重置比赛状态
- **200人同时提交答案时可能损坏成绩文件**:原来的写文件方式在高并发下有风险,改成排队串行写入

## 九点六、微信内置浏览器适配

参赛者答题页(`player.html`)针对微信内置浏览器做了这些处理:
- 禁止长按弹出复制/搜索菜单、禁止双指缩放误触、去掉点击时的高亮闪烁和300ms点击延迟
- 用 JS 动态计算真实可视高度,避免微信工具栏导致页面高度跳动或内容被遮挡
- 适配 iPhone 刘海屏底部安全区,避免按钮被系统手势条遮挡
- 姓名输入框仍可正常输入和选择文字(只禁用了非输入区域的文字选中)

## 九点七、计分规则怎么改,大屏幕说明文字是否会同步

**会自动同步。** 计分规则的两个参数现在放在 `config.js` 里:

```js
SCORE_BASE: 50,                 // 答对的基础分
SCORE_SPEED_BONUS_PER_SEC: 5,   // 剩余每秒钟的速度加成
```

改这两个数字,保存后在主持人控制台点"重新加载配置",实际计分逻辑和大屏幕底部显示的说明文字**都会用这两个新值**,不需要分别改两个地方——两者现在读的是同一份配置,不存在"改了公式但说明文字没跟着变"的问题。

如果想改的不是这两个参数,而是整个计分公式的计算方式(比如想改成阶梯式给分、或者不考虑速度只看对错),那还是需要改 `server.js` 里 `player:answer` 那部分的代码逻辑,这种结构性改动没法只靠配置文件解决,改完记得也去 `screen.html` 里把说明文字的拼接逻辑一并调整,让文字准确描述新公式。

## 九、安全性说明(内部活动够用,更高要求可以继续加固)

- 主持人页面的密码校验是服务端校验的,不是前端摆设,不会被绕过。大屏幕不再单独设密码,依赖"URL不公开+只能从主持人页面点开"来控制访问,如果需要更强的访问控制(比如担心有人猜到 `/screen.html` 这个地址直接打开),可以考虑给大屏幕也加回独立密码——需要的话告诉我可以再加上
- 正确答案只有在"公布答案"阶段之后才会通过网络下发给客户端,答题阶段客户端拿不到答案,判分也在服务端完成,避免了之前纯前端版本"打开控制台就能看到答案"的问题
- 如果部署到公网,建议再加一层 HTTPS(Render/Railway 默认自带),避免密码明文传输被截获
